# 代码复用审查与重构

把散落在各文件的重复逻辑抽取到统一的共享层（工具函数 / 常量 / 组件），消除"复制粘贴"式冗余。

## 何时触发

- 用户说"有没有重复代码""可复用""抽取组件/工具函数""代码冗余"
- 新功能开发前想先统一基础设施
- 定期消除技术债
- 审查某个改动时发现"这个逻辑好像在别处也写过"

## 核心原则

1. **先扫描再动手**：用 grep / wakita-scout 全局扫描重复模式，列出清单确认后再改
2. **纯重构不改行为**：不增删功能、不改 API 响应、不改路由。外部行为零变化
3. **单一真源**：每个概念只有一个权威定义处（如状态映射、平台配置），不允许多处定义
4. **薄包装可接受**：调用方如需空值兜底等差异，保留薄包装转发函数是合理的，不必强行消灭
5. **逐项独立交付**：每项抽取可单独验证（build 通过 + 运行正常），不要攒到最后一起测

## 工作流程

### 1. 扫描重复

用 grep 或 wakita-scout 并行扫描三类重复（具体命令模板见 `references/patterns.md`）：
- **重复逻辑片段**：同函数名/格式化/兜底模式在多处出现
- **重复数据定义**：同一份映射/常量/配置在多处定义
- **重复 UI 片段**：相同模板结构在多组件中复制

### 2. 列清单 + 定优先级

| 优先级 | 判断标准 |
|--------|---------|
| 高 | 3+ 处重复且定义冲突（逻辑不一致 → 潜在 bug） |
| 高 | 8+ 处模板片段完全相同 |
| 中 | 3-5 处重复但逻辑一致 |
| 低 | 2 处重复或语义略有差异 |

### 3. 抽取共享层

按优先级逐项抽取。**目标位置**：后端工具函数→`utils/`、前端工具函数→`utils/format.ts`、前端常量→已有的 config 文件、前端组件→`components/common/`。

```ts
// 函数抽取模板
export function sharedFunc(param: Type): ReturnType { ... }
```

```vue
<!-- 组件抽取模板 -->
<template>
  <span class="shared-component">
    <!-- 统一的模板结构，props 控制差异 -->
  </span>
</template>
<script setup lang="ts">
defineProps<{ key: string }>()
</script>
```

### 4. 替换调用处

每处改为 import 共享层 + 调用，删除本地定义。调用方如需空值兜底等薄包装，保留是合理的（不要强行消灭）。

### 5. 逐项验证

每项改完立即验证：前端 `npm run build`（0 TS 错误）、后端 `docker compose restart` + health check。

## 陷阱与注意事项

1. **同名不同义**：确认函数语义再合并，必要时改名而非硬并
2. **兜底来源不同**：抽取函数接收参数，保留调用处传不同字段
3. **截断长度差异**：共享函数用较短截断，需更长的外层补截断
4. **同步 vs 异步**：确认原调用是 `await` 还是同步，生成器类型要对应
5. **CSS 重复**：全局已定义的 class 别在组件局部再定义
6. **单位/文案统一**：同一格式化函数输出不一致时统一规范

## 完成验收

```bash
# 确认无残留重复——替换 pattern 为目标项目实际符号
grep -rn "重复函数名\|重复变量名" src/ | grep -v node_modules  # 预期空

# 构建 + 运行
npm run build                           # 前端：成功
docker compose restart && curl -sf ...  # 后端：200
```
