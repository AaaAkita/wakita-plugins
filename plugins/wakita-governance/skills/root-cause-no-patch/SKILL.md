# 根因优先，禁止补丁

修 bug 必须定位根因并根治。补丁/临时方案/降级容忍只是掩盖症状，问题会在别处以更难排查的形式复现，浪费更多时间。

## 何时触发

- 用户说"禁止补丁""修到正常为止""不要临时修正方案""根治"
- 同一个问题反复出现（第三次就是补丁信号）
- 遇到 `||` 降级、`try/catch` 吞错、`force` 重试等"让错误消失"的写法
- 报错被 `echo warning` 或 `|| true` 静默掉

## 核心原则

**补丁的特征**（出现以下任一即可疑）：
- 让错误"消失"而非解决：`cmd || echo warn`、`except: pass`、`|| true`
- 降级而非修复：装不上就跳过、连不上就换备选、功能缺失就隐藏入口
- 绕过而非定位："重启就好了""换个环境试试""加个超时跳过"
- 治标不治本：清缓存缓解、增大限制掩盖、加 retry 掩盖偶发崩溃

**根因修复的特征**：
- 能回答"为什么会崩"并指向具体原因
- 修复后同一根因的其他表现也一并消失
- 不依赖"重启""清缓存"等一次性操作

## 工作流程

### 1. 先理解为什么崩，再动手

不要急于"让它不报错"。先问：
- 这个错误的**直接原因**是什么？（如 esbuild 进程退出）
- 进程为什么退出？（socket 连接关闭 → `Socket.afterClose`）
- socket 为什么关闭？（容器内 esbuild 长驻服务不稳定）
- 为什么不稳定？（版本 bug / 环境不匹配 / 资源不足）

**追问 3-5 层"为什么"**，直到找到可操作的根因。

### 2. 区分"现象"和"根因"

| 现象（补丁治这个） | 根因（根治这个） |
|---|---|
| 页面 504 | esbuild 预构建崩 → 按需引样式触发二次预构建 |
| `The service was stopped` | esbuild 0.21.5 容器内 socket bug |
| 构建卡 13 分钟 | 缺 .dockerignore，COPY 1GB 本地目录 |
| Chrome 装失败 | apt 源用 http 被 80 端口干扰 |

**判断方法**：如果修了"现象"后换个环境/数据量/版本又会复现，那只是补丁。

### 3. 修复根因，验证根因消失

- 修复后，确认**根因链上每一环**都不再触发
- 不只看"不报错了"，要看"原来报错的路径现在正常走通"
- 用自动化验证（build/health/Playwright 控制台）而非"看起来好了"

## 真实案例：esbuild 504（追问 5 层）

```
现象：页面 504 Outdated Optimize Dep
  → 为什么？动态 import 的模块加载失败
    → 为什么？Vite 二次预构建时 esbuild 崩（The service was stopped）
      → 为什么崩？esbuild 0.21.5 长驻服务在 Docker 内 socket 不稳定
        → 为什么触发二次预构建？按需引组件样式运行时才被发现
          → 根因①esbuild 版本 bug ②按需引样式机制
```

**补丁路径**（错误，会复现）：
- 清 .vite 缓存 → 临时缓解，下次又 504
- 加 retry → 掩盖崩溃，偶发仍出现
- `optimizeDeps.include` 列组件 → 漏 loading/message，仍 504

**根因路径**（正确，彻底消失）：
- esbuild 升级到 0.28.1（修 socket bug）
- 样式改全量引入（消除二次预构建触发点）
- 验证：Playwright 控制台 0 错误，刷新多次稳定

## 真实案例：构建卡 13 分钟

```
现象：docker build 卡在 mkdir + chown 13 分钟
  → 为什么慢？chown -R 递归改 1GB 文件属主
    → 为什么有 1GB？COPY . . 把本地 venv(947MB) 拷进去了
      → 根因：缺 .dockerignore
```

**补丁**：加大超时、跳过 chown → 治标，每次构建仍慢
**根因**：加 .dockerignore 排除 venv → COPY 从卡死降到 2.5s

## 禁止清单

| 禁止 | 应该 |
|------|------|
| `cmd \|\| echo warn`（静默错误） | 修复 cmd 本身 |
| `except: pass`（吞异常） | 定位异常原因 |
| `--build-arg` 传代理重试 | 修网络/镜像源根因 |
| "重启容器就好了" | 找出为什么需要重启 |
| "清缓存试试" | 找出缓存为什么坏 |
| 降级容忍 `A \|\| fallback_to_B` | 修 A 或明确 B 是设计而非降级 |
| 加 retry 掩盖偶发崩溃 | 定位崩溃根因 |

**例外**：降级是设计意图时允许（如 patchright chromium 作为 chrome 的 fallback，且明确文档化）。但"装不上就跳过"不是设计，是掩盖。

## 验证标准

修完后问自己：
1. 我能一句话说清"为什么会崩"吗？不能 → 可能还是补丁
2. 去掉我的修复，问题会复现吗？不会 → 修了不该修的地方
3. 同一根因的其他表现也消失了吗？没有 → 只治了表象
4. 换个环境/数据量还会出吗？会 → 根因没除干净
