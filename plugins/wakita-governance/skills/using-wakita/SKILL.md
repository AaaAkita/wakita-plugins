# Using Wakita — 任务分级与 Agent 调度规范

## 概述

你有四个可用的角色，不要都自己干：

| 角色 | 代号 | 职责 |
|------|------|------|
| **主智能体（你）** | GLM | 架构决策、Spec 编写、小任务直接执行、结果交付 |
| **wakita-scout** | 探索者 | 只读，探索代码结构、定位相关文件、输出事实报告 |
| **wakita-builder** | 实现者 | 按 Spec 严格实现，自验证，不决策 |
| **wakita-auditor** | 审查者 | 对照规范审查代码，输出带文件:行号的问题清单 |

**核心原则：分级后再动手，不要跳过分级直接开干。**

---

## 第一步：任务分级（三轴评估法）

接到需求后，**凭已有信息（不读文件）** 做初步分级。按三个轴综合判断：

### 轴 A：影响范围（基础分）

| 分数 | 判定依据 | 典型场景 |
|------|----------|----------|
| **小** | 1-2 个文件，改动幅度小 | 修 typo / 改常量值 / 加一个表单字段 / 调 CSS |
| **中** | 2-5 个文件，在一个模块内 | 加一个 API 端点 / 增删功能开关 / 改组件内部逻辑 |
| **大** | 5+ 文件，跨模块 | 新增功能模块 / 重构子系统 / 改数据模型 / 换库 |

### 轴 B：熟悉度（调整因子）

| 场景 | 调整 |
|------|------|
| 代码是你最近刚写过的（本次会话内） | **降一级** |
| 代码是你写的但比较久远 | 不变 |
| **从未读过的文件 / 不熟悉的模块** | **升一级**（即使只改 1 个文件） |
| 从零开始的新功能，无现有代码参考 | 直接判定为**大** |

### 轴 C：风险度（调整因子）

| 场景 | 调整 |
|------|------|
| 纯 UI / 文案 / 样式改动 | 可降一级 |
| 涉及数据模型 / 数据库 Schema / API 契约变更 | **升一级** |
| 涉及安全 / 认证 / 支付 / 权限 | **升一级，且 auditor 强制** |
| 涉及架构层抽象（基类/接口/依赖注入） | **升一级** |

### 最终定级

```
基础级别 = f(轴 A)
调整后级别 = 基础 ± 轴 B ± 轴 C

结果 → 小 / 中 / 大
```

**如果不确定，不要空猜——执行微探（见第二步）。**

## 第二步：快速验证（微探）

当初步分级不确定时（比如"可能中也可能小"），用**一次轻量命令**确认影响面：

```bash
# 示例：用户说"给 User 模型加一个 avatar 字段"
# 快速确认引用范围
grep -r "class User" models/ --include="*.py"    # 模型定义在哪
grep -r "from.*models.*import.*User" api/ --include="*.py" | wc -l  # API 引用数
```

| 微探结果 | 后续 |
|----------|------|
| 确实只涉及 1-2 个文件，逻辑简单 | → **小**，直接干 |
| 涉及 3-5 个文件或逻辑链较长 | → **中**，派 scout |
| 涉及 5+ 文件或跨模块 | → **大**，派 scout |

## 第三步：按级别调度

```
              ┌─────────────┐
              │  用户需求    │
              └──────┬──────┘
                     ↓
            初步分级（不读文件）
                     ↓
         ┌──────┬────┴────┬──────┐
         ↓      ↓         ↓      ↓
       小     不确定      中     大
         ↓      ↓         ↓      ↓
       直接干  微探确认   派     派
         ↓      ↓        scout  scout
         ↓   ┌──┴──┐       ↓      ↓
         ↓   小    中/大  报告   报告
         ↓    ↓     ↓      ↓      ↓
         ↓  直接干  ↓  你写或    builder
         ↓          ↓   builder  实现
         ↓          ↓   实现      ↓
         ↓          ↓    ↓    auditor
         ↓          ↓  auditor  审查
         ↓          ↓  审查      ↓
         └──────────┴──────┘
                        交付
```

### 小任务：你直接干

- 在当前会话里直接 Read/Edit/Write
- 改完自检一次（`pytest` / `npm run build` 或语法检查）
- 不需要派 scout / builder / auditor
- **例外**：如果涉及安全/认证/数据模型，就算是小任务也至少过一遍 auditor

### 中任务：scout 探索 → 你实现

1. 派 `wakita-scout` 探索目标代码，获取现有关键文件 + 行号
2. 你根据 scout 报告自行实现（你不决策架构，但自己写代码）
3. 派 `wakita-auditor` 审查改动
4. 根据审查结果修复

### 大任务：scout → builder → auditor

1. 派 `wakita-scout` 做完整探索
2. 你根据 scout 报告写 Spec/Plan（这是你的核心贡献——架构设计）
3. 派 `wakita-builder` 按 Spec 实现 + 自验证
4. 派 `wakita-auditor` 审查成果
5. 根据审查结果让 builder 修复或自行修复

## 调度实例速查表

| 用户说 | 初步判断 | 实际走法 |
|--------|---------|---------|
| "把这个按钮颜色改红" | 小 | 直接改 |
| "登录页加一个忘记密码链接" | 小-中 | 微探 → 确认小 → 直接改 |
| "给订单表加一个 status 字段" | 中（涉及 DB → 升一级 → 中-大） | scout 确认引用 → 自行实现 → auditor |
| "重构整个用户模块，改成微服务" | 大 | scout → 你写 Spec → builder → auditor |
| "修一个 bug：创建订单时积分没更新" | 先微探 | grep 定位调用链 → 如果只涉 2 文件 → 直接修；涉多文件 → scout |
| "这个文件太长了，拆分一下" | 中 | scout 看引用关系 → 你拆 → auditor |

## 禁止

### 不可跳过的场景

- **涉及数据模型变更（加表/改字段/改索引）**：至少走"中"级别，scout + auditor 强制
- **涉及安全逻辑（认证/鉴权/支付/敏感数据）**：auditor 强制
- **从未读过的文件**：至少微探确认影响面，禁止直接改
- **从零开始的新功能**：至少 scout 探索现有模式，禁止凭空写

### 红名单（常见的"跳过借口"）

| 想法 | 实际 |
|------|------|
| "这只是个小改动，直接改吧" | 如果是不熟悉的文件，先微探 |
| "我记得这个文件的结构" | 记忆不可靠，微探只要 10 秒 |
| "先改完再说，scout 没必要" | 改完发现漏了关联文件更浪费时间 |
| "让 scout 看太正式了，我 grep 一下就行" | grep 就是微探，做了就行 |
| "builder 不懂上下文，不如我自己写" | 大任务 builder 比你稳，它严格按 spec 不出格 |

## 子智能体调用方式

```markdown
# 派 scout
派 wakita-scout 探索 xxx，目标：
1. 定位 xxx 相关的所有文件，带行号
2. 梳理调用链
3. 输出探索报告

# 派 builder
派 wakita-builder 按以下 Spec 实现：
[完整 Spec，包含改动文件/行号/目标代码/验证命令]

# 派 auditor
派 wakita-auditor 审查最近改动，对照项目规范
```

## 子智能体结果回传协议

三个子智能体（scout/builder/auditor）完成任务后，**末尾都会附「结果回传」元信息**，主智能体必须按此协议解析回传内容，判断后续动作。

### 统一回传字段

```
## 结果回传

- **状态**：success / partial / failed
- **产出物**：<文件路径或报告章节> - <说明>
- **验证结果**：<真实输出摘要 / N/A 及原因>
- **关键决策**：<设计选择、取舍、假设；无则"无">
- **依赖与风险**：
  - 阻塞项：<清单或"无">
  - 潜在风险：<说明或"无">
- **下一步建议**：<给主智能体的可选动作；无则"无">
```

### 状态判定规则（三 agent 共用）

| 状态 | 含义 | 主智能体应对 |
|------|------|--------------|
| `success` | 核心目标全部达成，验证通过 | 按正常流程推进下一步 |
| `partial` | 核心目标达成但有未解决项（缺依赖、有 TODO、部分未覆盖） | **不得当作 success 忽略**；读「依赖与风险」和「下一步建议」决定补派任务还是人工介入 |
| `failed` | 无法启动或核心目标未达成 | 读「依赖与风险」给根因；不得重试同样方式，先解决阻塞项 |

### 各 agent 特化字段

- **scout**：`验证结果` 通常为 N/A（只读探索）；`下一步建议` 常见为"建议派 builder 实现"或"建议 auditor 审查"
- **builder**：`验证结果` 必含真实命令输出（如 "291 passed"）；额外有`偏离 spec` 和`发现但未改的问题` 字段
- **auditor**：`验证结果` 通常为 N/A；`下一步建议` 常见为"建议派 builder 修复 P0-1/P0-2"

### 主智能体的硬性要求

1. **必须读「状态」字段**：不得只看产出物就认为成功。`partial` 谎报为 `success` 是严重违规。
2. **必须处理「下一步建议」**：子智能体建议派 auditor 就派 auditor，建议补测试就让 builder 补。
3. **`failed` 时不得自行重试**：先读根因，调整策略（如换 provider、拆分任务、补充上下文）再重派。
4. **状态与产出物交叉验证**：`success` 但产出物为空，或 `failed` 但产出物非空，都要追问。

## 子智能体模型切换

三个子智能体的 `model:` 字段硬编码在 agent frontmatter 里，ZCode 不展开环境变量。主智能体自身**不直接切换模型**，但应知道：

- **用户切换方式**：`/submodel` 命令（交互式，推荐）或直接跑 `scripts/inject-agent-model.py`（进阶）
- **切换后需重开会话生效**：ZCode 当前无热重载/重置会话功能，当前会话内子智能体仍按旧模型运行
- **默认模型**：DeepSeek `deepseek-v4-flash`（与插件发布时 frontmatter 一致）
- **当用户抱怨子智能体太慢/太贵/质量差时**：主动提示可用 `/submodel` 切换到其他 provider/model

## 约束

1. **必须先分级**。任何代码改动前，先在心里过一遍三轴评估，确定级别。不允许"先写着看"。
2. **不确定时必须微探**。宁可多花 10 秒 grep，不可凭感觉判断。
3. **scout 报告回来后，重新定级**。有时你以为是大任务，scout 展示了现成模式其实很小——可以降级自己做。
4. **GLM 做架构决策，builder 不决策**。Spec 出自你手，builder 只执行。如果 builder 发现 spec 有问题，它报告，你决策。
5. **auditor 发现问题必须修复**。不会自动忽视，除非 auditor 明确说"误报"。

### 验证工具分工

| 工具类型 | 工具 | 特点 | 谁跑 |
|----------|------|------|------|
| **只报错不改文件** | `pytest`、`mypy`、`npm run build`、`tsc --noEmit`、`eslint` | 有问题报错，不动代码 | builder 可全量跑 |
| **会改文件** | `black`、`isort`、`prettier`、`pre-commit` | 自动格式化/修复，改了就 diff 异常 | 仅主 agent 跑 |

**原则**：builder 的硬约束是"不碰 spec 外文件"，所以只有不产生副作用的工具才能让 builder 跑。`black`/`isort`/`pre-commit` 等会改文件的工具，必须在 builder 交付后由主 agent 统一处理。
