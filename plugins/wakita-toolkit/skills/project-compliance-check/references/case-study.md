## 修复过程

### 第 1 轮：致命清零（6 项）

1. 分布式锁续期 → TTL + buffer + 后台协程续期
2. docker-compose.yml 21 处密钥改 `${VAR:-default}`
3. 数据库/缓存端口 bind `127.0.0.1`
4. CORS allow_origins 改环境变量读取
5. 请求限流（登录 10/min, 注册 3/min, 高频操作 5/min）
6. 敏感数据加密密钥独立 + 弱密钥检测

### 第 2 轮：高危清零（4 项）

7. nginx SSL server block + certbot 自动续期 + CSP/HSTS 安全头
8. 告警通知：IM/邮件/短信三渠道
9. Prometheus metrics 端点 + 业务指标
10. 备份脚本（全存储）+ 每日 cron

### 第 3 轮：中危清零（18 项）

11. 后台任务 autoretry + 队列拆分
12. 数据库连接复用（不再每次重建 engine）
13. 外部请求 sleep 改为随机抖动
14. 坏数据记录 + 乐观锁
15. 全局状态 lifespan 清理 + 定期回收
16. 健康检查扩展覆盖所有依赖
17. WebSocket token 认证 + 心跳
18. RBAC 权限控制（敏感操作限管理员）
19. Token HttpOnly Cookie

## 关键经验

1. **密钥管理是第一步**——硬编码密钥进 git 后，改配置也救不回来，必须轮换
2. **锁续期常被忽略**——很多项目有锁但无续期，TTL 等于任务超时是定时炸弹
3. **告警通知是最容易被 stub 掉的功能**——开发时"以后再加"，上线后没人记得
4. **备份不是"有卷就行"**——Docker 卷损坏 = 全量丢失，必须有独立备份
5. **每个 `except: pass` 都可能丢数据**——至少记日志，最好保留坏数据到 dead letter
