# 项目合规筛查

新项目上线前按 6 个维度、28 个检查项逐项审计。每个检查项有明确的判定标准和三级修复优先级。

## 使用流程

1. 加载本 skill 后，先读取 `references/check-patterns.md` 了解各维度的检查方法
2. 用 wakita-scout 探索项目结构，按维度逐项判定
3. 输出含评分和优先级的筛查报告
4. 对不通过项附文件:行号证据和修复方案

## 检查维度

### 一、安全合规（7 项）

| # | 检查项 | 判定标准 | 致命 |
|----|--------|----------|:--:|
| 1.1 | 密钥管理 | 所有密钥通过环境变量/Secret Manager 注入，无硬编码在配置文件或 git 历史中 | ✓ |
| 1.2 | 密钥强度 | JWT/Session/Encryption 密钥 ≥ 32 字符随机值，非 `test-xxx`/`change-me`/空字符串 | ✓ |
| 1.3 | CORS 配置 | `allow_origins` 限具体域名列表，非 `["*"]`。若 `allow_credentials=True` 则不允许通配符 | ✓ |
| 1.4 | 传输加密 | 生产环境有 HTTPS/TLS 终止（nginx/caddy/CDN），配置 HSTS/CSP 安全头 | |
| 1.5 | 认证覆盖 | 业务端点均有认证中间件，公开端点（login/register/health）有明确注释 | |
| 1.6 | 权限分级 | 存在角色区分（admin/user），敏感操作（配置管理/数据删除/用户管理）限管理员 | |
| 1.7 | 网络隔离 | 数据库/缓存/消息队列端口不暴露到公网，或仅 bind 127.0.0.1 | ✓ |

### 二、并发与锁（4 项）

| # | 检查项 | 判定标准 | 致命 |
|----|--------|----------|:--:|
| 2.1 | 分布式锁 | 有锁则必须：① TTL 兜底 ② 持锁者校验 ③ TTL > 任务超时 + buffer | |
| 2.2 | 请求限流 | 关键端点有速率限制（登录/注册/触发类操作 ≤ 10次/分钟） | |
| 2.3 | 任务去重 | 定时/后台任务有防重复执行机制（锁/状态检查/幂等设计） | |
| 2.4 | 全局状态 | 模块级可变状态有生命周期管理（启动初始化 + 退出清理 + 定期过期回收） | |

### 三、数据完整性（5 项）

| # | 检查项 | 判定标准 | 致命 |
|----|--------|----------|:--:|
| 3.1 | 数据备份 | 有定期备份策略（日级）+ 异地/远程同步 + 保留周期（≥7天）+ 恢复演练 | ✓ |
| 3.2 | 失败重试 | 后台任务有自动重试（含退避策略），区分可重试/不可重试错误 | |
| 3.3 | 输入容错 | 外部输入（文件/API/消息）解析失败不静默丢弃，坏数据保留到死信队列供排查 | |
| 3.4 | 写入冲突 | 并发写入同一记录有乐观锁（version/updated_at）或悲观锁（SELECT FOR UPDATE） | |
| 3.5 | 连接池 | 数据库连接池有上限 + 超时 + 连接回收，session 有自动关闭保障 | |

### 四、可观测性（4 项）

| # | 检查项 | 判定标准 |
|----|--------|----------|
| 4.1 | 健康检查 | `/health` 检查所有关键依赖（DB/Cache/Queue），返回组件级状态 |
| 4.2 | 指标暴露 | 有 `/metrics` 端点（Prometheus 格式），含请求计数/延迟/业务指标 |
| 4.3 | 告警通知 | 关键事件（任务失败/服务降级/资源告警）有外部通知（IM/邮件/短信至少一种） |
| 4.4 | 日志规范 | 结构化日志（JSON），含 `timestamp`/`level`/`service`/`trace_id`，对接日志平台 |

### 五、外部依赖与网络（4 项）

| # | 检查项 | 判定标准 |
|----|--------|----------|
| 5.1 | 外部调用 | 所有外部 HTTP/RPC 调用有 timeout（≤30s）+ 重试（≤3次） |
| 5.2 | 频率控制 | 对外请求（API/爬虫）有差异化延迟 + 随机抖动，避免固定间隔 |
| 5.3 | 解析容错 | 输入解析失败不崩主流程，坏数据记录 + 告警 |
| 5.4 | 长连接管理 | WebSocket/SSE 有认证 + 心跳 + 断线重连 + 连接上限 |

### 六、部署与运维（4 项）

| # | 检查项 | 判定标准 |
|----|--------|----------|
| 6.1 | 资源限制 | 容器/进程有 CPU + 内存上限，防止单服务 OOM 影响整机 |
| 6.2 | 容器健康 | Docker/K8s 有 healthcheck，`depends_on` 用 `condition: service_healthy` |
| 6.3 | 优雅关机 | 退出时释放资源（DB连接/外部进程/临时文件），SIGTERM 不丢请求 |
| 6.4 | 单点风险 | 单实例服务有备份方案；或明确文档化单点接受声明 |

## 评分与定级

每个检查项判定为「通过 / 不通过 / 不适用」。最终评分 = 通过数 / (28 - 不适用数)。

| 评分 | 定级 | 含义 |
|------|------|------|
| ≥ 24 且无致命项 | 🟢 生产就绪 | 可上线 |
| 18–23 或致命项已清零 | 🟡 有条件就绪 | 补完剩余项后上线 |
| < 18 或有未修复致命项 | 🔴 不可上线 | 必须修完致命项 |

致命项（标记 ✓）：1.1 密钥管理、1.2 密钥强度、1.3 CORS、1.7 网络隔离、3.1 数据备份。
