# 项目基线检查

新项目启动时按 4 个维度、10 个检查项逐项审计。每个检查项有明确的判定标准，不过就修，修完就能交付，不搞大而全。

## 使用流程

1. 加载本 skill 后，先读取 `references/check-patterns.md` 了解各维度的具体搜索方法
2. 用 wakita-scout 探索项目结构，按维度逐项判定
3. 输出含评分的基线报告
4. 对不通过项附文件:行号证据和修复方案

## 检查维度

### 一、安全底线（3 项）—— 不做 = 数据泄露/服务被黑

| # | 检查项 | 判定标准 | 致命 |
|----|--------|----------|:--:|
| 1 | 密钥不入库 | 所有密钥走环境变量或 Secret Manager，不在配置文件中硬编码；`.env` 在 `.gitignore`；git 历史无明文密钥 | ✓ |
| 2 | 入口输入校验 | API/函数入口有空值、类型、长度校验；非法输入返回明确错误信息，不崩服务、不静默丢数据 | |
| 3 | 鉴权覆盖 | 业务接口有认证中间件；公开接口（login/register/health）有明确注释或文档说明无需鉴权 | |

### 二、数据底线（2 项）—— 不做 = 数据不一致/环境互串

| # | 检查项 | 判定标准 | 致命 |
|----|--------|----------|:--:|
| 4 | Schema as Code | 数据库表结构通过 Migration 管理，不手动改库；有单命令即可建表的脚本或工具 | ✓ |
| 5 | 环境隔离 | dev / prod 至少两套独立配置（数据库、缓存、第三方密钥不跨环境共用） | ✓ |

### 三、架构底线（3 项）—— 不做 = 改一处崩全身/排障无门

| # | 检查项 | 判定标准 |
|----|--------|----------|
| 6 | 前后端分离 | 前端和后端代码分目录，互不引用源码；通信仅走 API/HTTP，不混用服务端模板渲染 |
| 7 | 统一响应信封 | 所有 API 返回 `{ success, code, data, message }` 统一结构；有全局错误处理器；路由携带版本前缀（如 `/api/v1/`） |
| 8 | 异常不静默 | 有全局异常处理；catch 块至少记录完整错误栈，不 `pass`、不吞错、不静默返回成功 |

### 四、工程底线（2 项）—— 不做 = 别人跑不起来

| # | 检查项 | 判定标准 |
|----|--------|----------|
| 9 | 依赖可复现 | 有依赖锁定文件（`requirements.txt`/`package-lock.json`/`poetry.lock` 等），版本精确，不写 `latest` / `*` |
| 10 | README 可启动 | 克隆 → 安装 → 启动 三步能走通；含环境变量说明；含默认端口号 |

## 评分与定级

每个检查项判定为「通过 / 不通过 / 不适用」。最终得分 = 通过数 / (10 - 不适用数)。

| 得分 | 定级 | 含义 |
|------|------|------|
| 10/10 且致命项全通过 | 🟢 基线达标 | 可以交付，后续迭代可逐步引入 robust-architecture 规范 |
| 致命项全通过，非致命项有缺失 | 🟡 基本可用 | 建议补齐剩余项，但不阻塞交付 |
| 任一致命项不通过 | 🔴 不可交付 | 必须先修完致命项再交付 |

致命项（标记 ✓）：1 密钥不入库 / 4 Schema as Code / 5 环境隔离。

## 输出格式

```markdown
# 项目基线检查报告

**项目**: <名>  **日期**: <YYYY-MM-DD>  **得分**: X/10

| # | 检查项 | 状态 | 证据 |
|---|--------|:----:|------|
| 1 | 密钥不入库 | ✅ | `.env` 在 `.gitignore`，无硬编码密钥 |
| 4 | Schema as Code | ❌ | 未找到 migration 目录，`app/db.py:35` 直接 `CREATE TABLE IF NOT EXISTS` 建表 |

## 修复优先级
### 🔴 致命（交付前必修）
- #4 Schema as Code：引入 Alembic/Knex/Prisma Migrate，将裸 SQL 建表收敛到 migration 文件
### 🟡 建议修复（下次迭代）
- #2 入口输入校验：`routes/user.py:42` 未校验 `email` 格式
```

## 与现有 skill 的关系

```
project-baseline（10项）  →  robust-architecture（架构规范）  →  compliance-check（28项终审）
  项目第1天                     第1周逐步完善                      上线前
 "不出严重bug就行"             "开始规范化"                     "可以上线了"
```

- **project-baseline** 是本 skill，只查底线，10 分钟跑完
- **robust-architecture** 是配套规范，基线达标后逐步引入分层/API 标准化/Git 工作流
- **compliance-check** 是上线前终审，覆盖安全加固/并发/可观测/容器化等生产级要求
